================================================================================
CRYPTOGRAPHIC DESIGN
================================================================================

KEY DERIVATION (PBKDF2):
  Password + Invite Code Salt → 600,000 iterations → 256-bit Key
  
MESSAGE ENCRYPTION (AES-GCM):
  Plaintext + Random IV → AES-GCM(Key, IV, AAD) → Ciphertext + Auth Tag
  
IMAGE ENCRYPTION:
  Image Bytes → Strip EXIF → AES-GCM → Encrypted Blob
  
SESSION STORAGE:
  Keys stored in volatile memory only (globalThis)
  Auto-cleared on tab close, visibility change, or manual lock

================================================================================
DATABASE SCHEMA (4 TABLES)
================================================================================

chats:     Metadata only (user IDs, codenames, status, expires)
messages:  Encrypted blobs (iv, ciphertext, auth_tag, immutable)
images:    Encrypted blobs (iv, ciphertext, auth_tag, immutable)
invites:   Temporary pairing codes (expires after 24 hours)

All with Row-Level Security:
  - Users can only access their own chats
  - Users can only read/write their own messages
  - Messages are immutable (no updates)
  - Invite codes expire automatically

SECURITY FEATURES
================================================================================

ENCRYPTION:
  ✓ AES-GCM 256-bit (NIST Suite B)
  ✓ PBKDF2 with 600,000 iterations (NIST SP 800-132)
  ✓ Random IV per message (12 bytes, CSPRNG)
  ✓ Auth tag prevents tampering (16 bytes)
  ✓ No plaintext stored anywhere

KEYS & SESSIONS:
  ✓ Keys derived from password, never transmitted
  ✓ Keys stored only in memory (non-extractable)
  ✓ Keys auto-cleared on tab close/visibility change
  ✓ No localStorage, no sessionStorage for keys
  ✓ Each chat has independent key (no master key)

AUTHENTICATION:
  ✓ Supabase anonymous auth (each session unique UID)
  ✓ RLS policies enforce chat membership
  ✓ Codename-based identity (no personal info)
  ✓ Invite codes expire after 24 hours

TRANSPORT:
  ✓ HTTPS enforced (Vercel auto-redirects)
  ✓ TLS 1.3 minimum (Vercel enforces)
  ✓ No unencrypted credentials in transit
  ✓ CSP headers configurable

================================================================================
THREAT MITIGATIONS
================================================================================

Man-in-the-middle:      HTTPS + TLS 1.3
Server compromise:      AES-GCM encryption (no keys)
Database breach:        Ciphertext + PBKDF2 600k iterations
Weak password:          High iteration count + user responsibility
XSS injection:          Input validation, CSP headers
Nonce reuse:            Random IV per message (guaranteed unique)
Message tampering:      AES-GCM auth tag
Replay attacks:         AAD includes timestamp
Invite code exposure:   24-hour expiration
Session hijacking:      Auto-lock on tab visibility change

================================================================================
USER FLOWS
================================================================================

CREATE CHAT (User A - "Mr. Blue"):
  1. Click "CREATE CHAT"
  2. Enter codename
  3. System generates invite code (32 random bytes)
  4. Share code via secure channel
  5. Wait for User B

JOIN CHAT (User B - "Mr. Green"):
  1. Click "JOIN CHAT"
  2. Paste invite code
  3. Enter codename
  4. Chat becomes active

UNLOCK & MESSAGE (Both):
  1. Enter unlock secret (password)
  2. Key derived: PBKDF2(password, salt)
  3. Messages decrypt in browser memory
  4. Send/receive messages
  5. Upload encrypted images

LOCK (Manual or Automatic):
  1. Click lock button OR
  2. Tab hidden OR
  3. Window closed
  4. Keys cleared from memory
  5. Refresh = locked again
